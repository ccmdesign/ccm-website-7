# Varro Content Sync & Marketing File Integration

## Problem Frame

Blog posts are generated by the Varro project, which produces two files per article: a blog post (`final.md`) and a marketing file (`marketing.md`) with social media copies for LinkedIn, X, Mastodon, etc. Currently these files are manually copied into the CCM website repo and the marketing files are discarded. There is no automated sync, no place for marketing files in this repo, and no way for the admin UI to surface the LinkedIn draft copy when distributing a post.

## Requirements

### Sync Command

- R1. A CLI script (`scripts/sync-varro.ts`) copies articles from Varro's content archive into this repo.
- R2. For each Varro archive folder, the script copies:
  - `final.md` → `content/blog/{slug}.md` (with frontmatter migrated to published schema)
  - `marketing.md` → `content/marketing/{slug}.md` (copied as-is, preserving all platform copies)
- R3. Frontmatter migration on `final.md` follows the same rules applied to the 22 drafts:
  - `category` (singular) → `categories` (array)
  - `keywords` + `primary_keyword` → `seo_tags`
  - `excerpt` → `tldr` (copy)
  - `author` → "CCM Design Team"
  - Drop: `meta_title`, `meta_description`, `stage`, `status`, `related_posts`, `cta`
  - Add blank: `brow: ''`, `tagline: ''`, `tags: []`
- R4. The script assigns `date` values following the Mon/Thu posting cadence, continuing from the last scheduled post in `content/blog/`.
- R5. Posts that already exist in `content/blog/` (matched by slug) are skipped by default. A `--force` flag overwrites existing posts.
- R6. The script accepts a `--dry-run` flag that prints what would be synced without writing files.
- R7. The Varro content archive path is configurable via a `--source` flag, defaulting to a sensible relative path or env var (`VARRO_CONTENT_ARCHIVE`).

### Marketing File Storage

- R8. Marketing files live in `content/marketing/` as individual `.md` files named by slug (e.g., `content/marketing/senior-talent-opportunity-cost.md`).
- R9. A blog post and its marketing file are linked by matching slugs — no frontmatter cross-reference needed.
- R10. Marketing files are not part of the blog Nuxt Content collection. They either belong to a separate `marketing` collection or are read directly from the filesystem by server utils.

### Admin UI Integration

- R11. When the admin UI shows a blog post's LinkedIn actions, it checks for a matching marketing file at `content/marketing/{slug}.md`.
- R12. If a marketing file exists, the admin UI displays a "LinkedIn Draft" preview showing the LinkedIn long-form post content extracted from the marketing file.
- R13. When the user clicks "Draft" on a post with a marketing file, the LinkedIn service receives the long-form post content from the marketing file (not manually composed).
- R14. If no marketing file exists for a post, the LinkedIn draft flow falls back to the current behavior (using excerpt or manual input).

## Success Criteria

- Running `npx tsx scripts/sync-varro.ts --source /path/to/archive` copies new posts and marketing files with correct frontmatter and dates
- Marketing files are accessible from the admin UI by slug
- Clicking "LinkedIn Draft" in admin for a synced post shows the correct LinkedIn copy from the marketing file
- Existing posts are not overwritten unless `--force` is used
- The blog frontend is unaffected — marketing files don't appear as blog posts

## Scope Boundaries

- Sync is manual (CLI), not automated or triggered by CI
- Only LinkedIn content is surfaced in the admin UI for now; other platforms (X, Mastodon, etc.) in the marketing file are stored but not displayed
- The sync script lives in this repo, not in Varro
- Varro's internal archive structure (hash-named folders) is a read-only input; this project doesn't modify Varro files
- The marketing file format is owned by Varro; this project reads it as-is

## Key Decisions

- **Separate files, not frontmatter**: Marketing content lives in `content/marketing/{slug}.md` rather than embedded in blog post frontmatter. This avoids bloating blog frontmatter with multi-platform marketing copies and keeps the Varro output portable.
- **Slug-based linking**: No explicit cross-reference field needed. The slug is the join key between blog post and marketing file.
- **Frontmatter migration in sync**: The sync script handles schema translation so Varro doesn't need to know about the CCM website's frontmatter schema.
- **Mon/Thu date assignment**: The sync script auto-assigns dates following the established 2x/week cadence, removing manual scheduling work.

## Dependencies / Assumptions

- Varro's `final.md` always has a `slug` field in frontmatter
- Varro's `marketing.md` always has an `article_slug` field matching the blog post's slug
- The marketing file's LinkedIn section follows a consistent markdown structure (## Professional Posts → ### LinkedIn → #### Long-Form Post)
- The existing content distribution plan (2026-03-24) is updated to read LinkedIn content from marketing files instead of frontmatter
